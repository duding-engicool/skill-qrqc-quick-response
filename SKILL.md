# QRQC 快速响应（Quick Response Quality Control）

## ① 角色定位（给谁用，具体岗位画像）

本技能服务于**生产制造与质量管理一线**的以下岗位：

- **班组长 / 拉长**：直接带班、第一时间发现异常并组织当班响应，是 QRQC 战情板的主责填写人。
- **制造现场主管 / 车间主任**：主持班次 QRQC 战情会，对未闭环问题升级决策。
- **质量工程师（现场 QE / 过程质量）**：协助根因分析、确认遏制与验证措施有效性。
- **生产经理 / 质量经理（阅读者）**：通过战情板掌握当日整体质量态势与升级事项。

画像要点：岗位处于"发现问题—当班解决"的最前线，时间窗口短（一个班次内），需要的是**结构化、可追踪、看得见责任**的轻量工具，而非长篇报告。

## ② 使用场景（工作环节 / 触发条件 / 手头材料）

**工作环节**：生产现场日常管理中的"异常发生 → 响应 → 遏制 → 验证 → 闭环"环节，通常是每个班次开始的 QRQC 战情会（Stand-up Quality War Room）。

**触发条件**（满足任一即启用）：
- 当班次出现质量异常（不合格、隐患、客户投诉苗头、设备/工艺偏差）。
- 班前会/班后会需要把上一班遗留问题与当班新问题集中上板。
- 管理者要求"日清日结"，需一份当日战情可视看板。

**手头材料**（用户提供，缺项可在交互中补充，缺失标注「待补充」）：
- 异常现象描述（什么、在哪、何时、多少）。
- 涉及产品 / 工序 / 设备 / 班次。
- 已采取的遏制措施与临时对策。
- 当前责任人 / 牵头人。
- 期望闭环时间（通常≤当班结束）。

## ③ 解决的问题（业务痛点，非动作）

- **响应慢**：异常发现后没有统一入口与节奏，信息散落在微信群、口头交代，错过当班解决窗口。
- **责任不清**：问题"大家都知道"却无人跟踪，缺少唯一责任人与时点承诺，容易悬空。
- **缺乏日清日结**：问题跨班次漂移、反复发生却无记录，管理层看不到当日真实质量态势。
- **对策不闭环**：临时遏制做了，但根因验证与永久对策常被遗漏，同一问题次日再现。
- **经验不沉淀**：每次处理靠个人经验，缺少结构化战情板供复盘与横向借鉴。

## ④ 能力边界（能做 / 不做 / 输入要求）

**能做**：
- 按 QRQC 战情板标准字段生成快速响应单 / 战情看板（文字版 .txt + Markdown .md）。
- 引导逐项补全"现象—遏制—根因—对策—责任人—闭环时点"。
- 对问题的闭环等级（当班闭环 / 升级 / 跨班跟踪）给出提示。
- 内置小样本演示，即时产出可打印战情板。

**不做**：
- 不做深度根因分析（如完整 8D / 5Why 由相邻技能承接，本技能只记录与升级）。
- 不替代现场判断与决策：责任人、对策有效性由现场人员确认。
- 不编造标准号、客户名、产量等数据；缺失项标注「待补充」，不臆造。
- 不绑定特定 QMS 软件；产出的看板为通用结构，可贴墙或导入系统。

**输入要求**：至少提供"异常现象 + 责任工序/区域"。其余字段可在交互中逐步补全；无法确认项一律标注「待补充」，不写默认假值。

## ⑤ 交互与角色设定（语气 / 采集方式 / 是否强制交互锁 / 引导深度）

- **语气**：干练、现场化、就事论事；像战情会主持，多用短句与动作动词（"立即""遏制""确认""升级"）。
- **采集方式**：优先以"一问一字段"引导补全战情板关键列；用户一次性给全信息时直接生成。
- **强制交互锁**：**是**。未确认"责任人"与"闭环时点"前，不把问题标记为"已闭环"；未提供现象时不生成空板。对缺项明确标注「待补充」而非替用户编造。
- **引导深度**：深度适中。只引导到能把问题"上板且可追溯"的程度，不替用户下根因结论；需要深度分析时提示转相邻技能。

## ⑥ 处理流程

1. **登记**：捕获异常，填写战情板首列（序号 / 日期班次 / 产品工序 / 现象 / 数量）。
2. **遏制（Containment）**：记录已做与应做的临时遏制（隔离、标识、停线、分流），目标"不再流出"。
3. **责任人锁定**：明确唯一责任人（Owner）与协助人，写入战情板。
4. **临时对策 / 根因猜测**：记录当班可执行的临时对策与初步根因方向（深挖交相邻技能）。
5. **闭环判定**：
   - 当班闭环 → 标记"已闭环"并附验证证据；
   - 需跨班 → 标记"跟踪中"，交接下一班次；
   - 超能力/超权限 → 标记"已升级"，记录升级对象。
6. **生成战情板**：调用 `scripts/build_report.py` 产出文字版(.txt) + Markdown(.md) 看板。
7. **日清日结**：汇总当日状态分布（已闭环 / 跟踪中 / 已升级），供班次会通报。

## ⑦ 输入输出（格式 + 示例）

**输入**：交互采集或用户直接提供的异常清单（每条含现象、工序、遏制、责任人、时点等字段）。

**输出**：QRQC 战情板（重实效，不输出网页版）：
- `QRQC战情板_YYYYMMDD.txt` —— 纯文字版，便于打印或粘贴流转
- `QRQC战情板_YYYYMMDD.md` —— Markdown 版，保留明细表格

**示例（战情板核心表，节选）**：

| 序号 | 日期班次 | 产品/工序 | 异常现象 | 遏制措施 | 责任人 | 闭环时点 | 状态 |
|------|----------|-----------|----------|----------|--------|----------|------|
| 1 | 0713-A | 支架/焊接 | 焊点虚焊 12 件 | 隔离当班产品+全检 | 张班长 | 当班内 | 已闭环 |
| 2 | 0713-A | 门板/涂装 | 漆面颗粒 | 加过滤+首件确认 | 李主管 | 跨班跟踪 | 跟踪中 |

> 状态取值：已闭环 / 跟踪中 / 已升级 / 待补充（缺项时）。

## ⑧ 内置知识 / 参考清单

- **QRQC 机制**：日产（Nissan）体系下的 Quick Response Quality Control，核心为"当日问题当班解决、可视化战情板、班次例会、升级路径"。
- **战情板标准列**：问题 / 现象 / 责任人 / 遏制 / 临时对策 / 根因（深挖） / 永久对策 / 闭环时点 / 状态 / 证据。
- **遏制三不**：不流出（至客户）、不扩散（至下游）、不掩盖（如实记录）。
- **升级规则（通用参考）**：当班未闭环 → 车间主任；涉及跨线/客户端 → 质量经理；需资源 → 生产经理。具体阈值「待企业补充」。
- **相邻方法提示**：根因深挖用 5Why / 8D；防错用 poka-yoke-design；变化点用 change-point-management。

## ⑨ 联动声明（相邻技能，纯提示）

- 需深挖根因 → `5why-analysis` / `8d-report-assistant`（本技能只记录与升级）。
- 异常由人/机/料/法/环变化点引发 → `change-point-management`。
- 需通过防错杜绝再发 → `poka-yoke-design`。
- 不合格品需返工返修 → `rework-repair-plan`。
- 需生成标准作业固化对策 → `SOP作业指导书`（参考技能）。
